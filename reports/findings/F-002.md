## Finding #F-002 — API Status Endpoint Exposes Internal Application Details (gopps.global.com)

**Date Found:** 2026-03-12
**Tester:** Baskar
**Engagement:** Multi-Target Penetration Test

---

### 1. Overview

| Field | Details |
|---|---|
| **Vulnerability Type** | Information Disclosure |
| **Severity** | Medium |
| **CVSS Score** | 5.3 |
| **CVSS Vector** | CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N |
| **OWASP Category** | A05:2021 – Security Misconfiguration |
| **Affected Component** | /api/status, /api/status/basic, /api/about |
| **Authentication Required** | None |

---

### 2. Description

The `/api/status` endpoint on gopps.global.com returns detailed internal application information without requiring authentication. The response includes:

- Application name and exact build version (`1.0.364.0`)
- Fields for database connection string, ElasticSearch configuration, and security configuration (currently null for unauthenticated users but field names are exposed)
- Complete user privileges/permission model structure revealing all authorization capabilities in the system
- Debug flag (`local: "true"`) suggesting development configuration in production

The exposed privilege model includes: `canAccessAdmin`, `canViewOtherCompanyRedactedDetails`, `canAccessGPlanOrders`, `canCreateOrderFromGPlan`, `canMaintainOwnOrder`, `canMaintainOtherOrders`, `canViewPropertiesOnOtherCompanyBuyingAreas`, `canCreateGOppsOrder`, `canCreateOption`.

---

### 3. Impact

**Business Impact:**
The exposed permission model reveals the complete authorization structure of the application, helping an attacker understand what privilege escalation targets exist. The field names for database and ElasticSearch configurations indicate these could be populated for authenticated users, potentially leading to credential exposure.

**Technical Impact:**
- **Confidentiality:** Medium — Internal architecture, version, and authorization model exposed
- **Integrity:** None
- **Availability:** None

---

### 4. Affected Endpoints / Parameters

| # | Method | Endpoint | Notes |
|---|---|---|---|
| 1 | GET | /api/status | Full status with privilege model and config fields |
| 2 | GET | /api/status/basic | Same as /api/status |
| 3 | GET | /api/about | Version, app name, debug flag |

---

### 5. Steps to Reproduce

1. Send an unauthenticated GET request:
   ```bash
   curl -s "https://gopps.global.com/api/status"
   ```

2. Observe the response containing application details, privilege model, and configuration field names.

---

### 6. Evidence

**HTTP Response (/api/status):**
```json
{
  "applicationName": "gOpps",
  "version": "1.0.364.0",
  "internalAssemblies": null,
  "externalAssemblies": null,
  "databaseConnectionString": null,
  "elasticSearchConfiguration": null,
  "securityConfiguration": null,
  "userDetails": {
    "userPrivileges": {
      "canAccessAdmin": false,
      "canViewOtherCompanyRedactedDetails": false,
      "canAccessGPlanOrders": false,
      "canCreateOrderFromGPlan": false,
      "canMaintainOwnOrder": false,
      "canMaintainOtherOrders": false,
      "canViewPropertiesOnOtherCompanyBuyingAreas": false,
      "canCreateGOppsOrder": false,
      "canCreateOption": false
    }
  }
}
```

**HTTP Response (/api/about):**
```json
{
  "version": "1.0.364.0",
  "application": "gOpps",
  "local": "true"
}
```

---

### 7. Root Cause Analysis

The status and about endpoints are designed for monitoring/debugging but are exposed without authentication in production. The `local: "true"` flag in `/api/about` strongly suggests development configuration is active in the production environment.

---

### 8. Remediation

**Primary Fix:**
- Restrict `/api/status`, `/api/status/basic`, and `/api/about` endpoints behind authentication
- Remove sensitive field names (databaseConnectionString, elasticSearchConfiguration, securityConfiguration) from the response entirely, or only expose them to admin-level users
- Set `local` to `false` in production configuration

**Secondary Controls:**
- Create a minimal health check endpoint that returns only UP/DOWN status without internal details
- Ensure development/debug configurations are never deployed to production environments

---

### 9. Retest Criteria

This finding is remediated when:
- [ ] `/api/status` returns 401/403 for unauthenticated requests
- [ ] No internal configuration field names or privilege models are exposed without authentication
- [ ] `/api/about` does not expose the `local` debug flag
