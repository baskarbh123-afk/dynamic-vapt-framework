## Finding #F-005 — XMLRPC Enabled with system.multicall on WordPress (production.victoriousfestival.co.uk)

**Date Found:** 2026-03-12
**Tester:** Baskar
**Engagement:** Multi-Target Penetration Test

---

| **Vulnerability Type** | Security Misconfiguration |

| **OWASP Category** | A05:2021 – Security Misconfiguration |
| **Affected Component** | /xmlrpc.php |
| **Authentication Required** | None |

---

### 2. Description

The XMLRPC interface is enabled on production.victoriousfestival.co.uk with the full WordPress method set including `system.multicall`. While the `wp-cron.php` is also publicly accessible on this site, the primary concern is that `system.multicall` allows an attacker to bundle multiple authentication attempts in a single HTTP request, effectively amplifying brute-force attacks.

Available methods include: `system.multicall`, `system.listMethods`, `pingback.ping`, `wp.getUsersBlogs`, `wp.getUsers`, `metaWeblog.newPost`, `metaWeblog.editPost`, `blogger.newPost`, and many more.

**Additionally affected:** wp-cron.php is publicly accessible on:
- fellasstudios.com
- production.victoriousfestival.co.uk
- www.victoriousfestival.co.uk

---

### 3. Impact

**Business Impact:**
An attacker could use system.multicall to attempt hundreds of password combinations in a single request, bypassing rate limiting that operates on a per-request basis. The pingback.ping method could potentially be leveraged for SSRF (though testing showed "Invalid discovery target" response).

**Technical Impact:**
- **Confidentiality:** None
- **Integrity:** Low — Potential for brute-force amplification leading to credential compromise
- **Availability:** None

| # | Method | Endpoint | Notes |
|---|---|---|---|
| 1 | POST | /xmlrpc.php | Full method list including system.multicall |
| 2 | GET | /wp-cron.php | Publicly accessible (200 on 3 sites) |

---

### 5. Steps to Reproduce

1. Verify XMLRPC is enabled and list methods:
   ```bash
   curl -s -X POST "https://production.victoriousfestival.co.uk/xmlrpc.php" \
     -H "Content-Type: text/xml" \
     -d '<?xml version="1.0"?><methodCall><methodName>system.listMethods</methodName></methodCall>'
   ```

2. Observe the response contains system.multicall and all wp.* methods.

---

### 6. Evidence

**HTTP Response (truncated):**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<methodResponse>
  <params>
    <param>
      <value>
        <array><data>
          <value><string>system.multicall</string></value>
          <value><string>system.listMethods</string></value>
          <value><string>pingback.ping</string></value>
          <value><string>wp.getUsersBlogs</string></value>
          <value><string>wp.getUsers</string></value>
          ...30+ methods
        </data></array>
      </value>
    </param>
  </params>
</methodResponse>
```

---

### 7. Root Cause Analysis

XMLRPC is enabled by default in WordPress. The iThemes Security plugin is installed but has not been configured to disable XMLRPC. The `system.multicall` method is inherent to the XML-RPC specification and cannot be individually disabled without disabling XMLRPC entirely.

---

### 8. Remediation

**Primary Fix:**
Disable XMLRPC entirely if not required:
- Via iThemes Security settings (already installed) → Enable "Disable XML-RPC"
- Or via `.htaccess`: `<Files xmlrpc.php> Order deny,allow Deny from all </Files>`
- Or via `functions.php`: `add_filter('xmlrpc_enabled', '__return_false');`

**Secondary Controls:**
- Disable public access to wp-cron.php and use server-side cron instead
- Implement Web Application Firewall rules to block XMLRPC brute-force attempts
- Enable account lockout after failed login attempts
