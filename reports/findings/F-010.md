**Status:** DRAFT

### 2. Description

The self-hosted Jitsi Meet instance at meet.geizhals.at exposes its full configuration via /config.js, revealing:

- XMPP domain: meet.geizhals.at
- MUC domain: conference.meet.geizhals.at
- BOSH endpoint: //meet.geizhals.at/http-bind
- Token-based roles are disabled (enableUserRolesBasedOnToken: false)
- Pre-join page is enabled but rooms can be created by anyone

Additionally, any URL path on meet.geizhals.at returns HTTP 200, meaning meeting rooms can be created without authentication. An unauthorized user could create rooms with misleading names (e.g., "board-meeting", "hr-review") for phishing or social engineering.

---

### 3. Impact

**Business Impact:** Unauthorized room creation enables phishing — attackers can create legitimate-looking meeting URLs under the company domain for social engineering attacks.

**Technical Impact:**
- **Confidentiality:** Low — Internal XMPP infrastructure details exposed
- **Integrity:** Low — Unauthorized meeting room creation
- **Availability:** None

---

### 4. Steps to Reproduce

1. View config: `curl -s "https://meet.geizhals.at/config.js" | head -30`
2. Create room: Navigate to `https://meet.geizhals.at/any-room-name` — returns HTTP 200

---

### 5. Remediation

- Require authentication for room creation (configure Prosody with token-based or LDAP auth)
- Set enableUserRolesBasedOnToken to true
- Restrict config.js exposure by serving a minimal version
